# Convert Pro 3 / File Converter 병합 고민 정리

## 배경

- 현재 `Convert_pro3`와 `File_converter`는 서로 다른 서버에서 운영 중이다.
- 두 프로그램의 변환 엔진 성격이 다르다.
  - `Convert_pro3`: CSV 중심 파이프라인
  - `File_converter`: 485(`.txt`) 파싱/변환 중심
- 사용자 관점에서는 웹에서 업체/현장/파일이 통합되어 보이길 원한다.

## 지금까지 확인한 사실

- 단순 코드 병합은 리스크가 크다.
  - 변환 파이프라인 영향 범위가 넓고 회귀 위험이 높다.
  - 장애 발생 시 원인 분리가 어려워진다.
- 반면 목록/메타 연동은 상대적으로 안전하다.
  - UI 통합 효과가 빠르게 나온다.
  - 엔진은 분리된 채로 유지 가능하다.

## 핵심 쟁점

### 1) 서버를 하나로 볼 것인가?

- 사용자 관점:
  - 한 웹에서 모두 보이면 사실상 "하나처럼" 느껴진다.
- 운영 관점:
  - 실제 프로세스/서버는 분리 유지가 더 안전하다.

결론: 당장은 "표시 통합", "엔진 분리"가 적절하다.

### 2) 설정(config)을 공유할 것인가?

- 완전 실시간 공유:
  - 동기화 충돌, 결합도 증가, 장애 전파 위험
- 완전 분리:
  - 업체/현장명이 어긋날 수 있음

권장:

- 설정 파일은 분리 유지
- 업체/현장 마스터만 공통 관리(웹 또는 마스터 API)

### 3) 485 구분을 확장자로만 할 것인가?

- `.txt`만 기준으로 분리하면 단기적으로는 단순하지만 불안정하다.
- 권장 분리 기준:
  1. `source` (`local`, `external_485`)
  2. 확장자(`.csv`, `.txt`)
  3. 필요 시 경로 규칙

## 현실적인 운영 시나리오 (추천)

## 시나리오 A: 프로그램 2개 유지 + 마스터 데이터 통일

- `Convert_pro3`와 `File_converter` 엔진은 각각 유지
- 웹에서 업체/현장 생성은 한 곳에서만 수행
- 각 프로그램은 업체/현장 목록을 읽어와서 사용
- 파일 등록/변환은 각 서버에서 별도 수행

장점:

- 변환 안정성 보존
- 책임 분리 명확
- 점진적 개선 가능

단점:

- 프로그램 2개 운영 필요

## 시나리오 B: 완전 코드 병합

- 485 엔진과 CSV 엔진을 한 코드베이스로 통합
- 서버/배포도 단일화

장점:

- 장기적으로 구조 단순

단점:

- 초기 리스크 매우 큼
- 회귀 테스트 비용 큼
- 장애 시 영향 범위 큼

## 현재 권장 결론

- 단기: **시나리오 A**
  - 프로그램 2개 유지
  - 업체/현장 마스터 통일
  - 웹 표시 통합
- 중장기:
  - 운영 데이터/테스트가 충분히 쌓인 뒤 시나리오 B 검토

## 실행 단계 제안

1. 업체/현장 마스터 스키마 확정
  - 키: `company`, `site`, `site_id`(권장)
2. 웹에서 업체/현장 생성 단일화
3. 두 프로그램은 마스터를 읽어 UI 선택지에 반영
4. 파일 등록은 각 프로그램에서 수행
5. 웹에서 `source` 기반으로 목록/상세 분기
6. 안정화 후 병합 재평가

## 결정이 필요한 항목

- 업체/현장 마스터의 단일 소스 위치
  - 웹 API
  - 공용 JSON
- 동기화 방식
  - 수동 동기화 버튼
  - 시작 시 자동 pull
- 충돌 정책
  - 이름 충돌 시 `company+site` 복합키 사용 여부

## 메모

- "빨리 통합"보다 "안전하게 통합"이 장기 비용을 줄인다.
- 현재 단계에서는 변환 엔진 병합보다 운영/데이터 모델 정렬이 우선이다.
